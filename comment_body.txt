Hi @Chetan0e, thanks for getting back to me. That's a very fair point! I've dug into the codebase to find concrete examples of where the current error handling approach falls short and could cause issues:

### 1. Silent Failures & Security Risks (e.g., `auth.py`)
Because exceptions are currently handled locally on a per-route basis, there are instances where broad exception handling masks critical failures. For example, in `backend/app/routes/auth.py`'s `/refresh` endpoint, the code attempts to blacklist the old refresh token but wraps it in:
```python
    except (JWTError, Exception):
        pass  # Token was already validated above; decoding won't fail here
```
If `get_db()` fails or the MongoDB `insert_one` operation fails due to network issues, the exception is silently swallowed. The old token is **not** blacklisted, but the system still proceeds to issue a new token pair. This creates a vulnerability where token rotation fails silently, allowing potential replay attacks. A global exception handler would prevent developers from having to make these risky local `try/except` decisions.

### 2. Massive Boilerplate & Duplication (e.g., `pipeline_routes.py`)
Currently, almost every single endpoint in `pipeline_routes.py` (and others) wraps its logic in the exact same boilerplate:
```python
    try:
        # business logic
    except HTTPException:
        raise
    except Exception as e:
        logger.error(f"Error ...: {e}", exc_info=True)
        raise HTTPException(status_code=500, detail="Failed to ...")
```
This forces developers to manually remember to log and return a 500 error every time they write a new endpoint. It clutters the codebase and violates DRY principles.

### 3. Risk of Leaked Exceptions
If a contributor adds a new endpoint but forgets to include the repetitive `try/except` block (which is easy to do), any unexpected error (like a `KeyError` or a database timeout) will bubble up directly to Uvicorn/FastAPI's default handler. This could potentially leak internal stack traces to the client and result in inconsistent API response formats.

**Proposed Scope:**
Implementing a global `@app.exception_handler(Exception)` at the FastAPI application level would allow us to:
1. Strip out the repetitive `try/except Exception` blocks from all route files, making the code much cleaner and easier to read.
2. Ensure every unexpected error is consistently logged with context.
3. Guarantee a standard JSON structure for all 500 errors across the entire API, preventing silent failures like the one in the auth route.

Let me know if this provides enough justification to prioritize this architectural improvement! I'm happy to put together a PR for this so you can see how much cleaner the routes become.
